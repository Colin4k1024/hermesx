关键设计原则总结

// FIGURE 9 — Security & Isolation Design Matrix

身份 & 数据隔离
维度实现机制安全保证
租户身份 API Key → DB 查询
JWT → claims 提取		 TenantID 来自凭证, 不信赖 header
数据隔离 tenant_id FK + 索引 查询自动按租户过滤
技能隔离 MinIO prefix="{tenantID}/" ListObjects 限定 prefix
记忆隔离 WHERE tid=? AND uid=? 上下文不跨租户
会话隔离 (platform, chat_id, tenant) session key 三元组唯一
人格隔离 {tenantID}/SOUL.md AI 行为因租户而异
运维 & 合规
维度实现机制安全保证
认证 Extractor Chain
(Static / JWT / APIKey)		 短路模式, 任一成功即终止
权限 RBACMiddleware
admin / user / operator		 API Key roles[] 控制
限流 RateLimitMiddleware
RPM 租户级限流		 动态按 tenants.rate_limit_rpm
审计 AuditMiddleware
+ audit_logs 表		 记录 tenant_id + user_id + action
Agent 缓存 per-session agent pool prompt cache 复用 agent 实例
GDPR CASCADE DELETE
/gdpr/export		 租户删除时全量数据清除
核心安全承诺
✓ TenantID 绝不 从请求 header 获取 ✓ 所有 PostgreSQL 查询强制包含 tenant_id ✓ MinIO ListObjects 限定 prefix = tenantID ✓ API Key hash 存储, SHA-256 校验 ✓ 租户删除 CASCADE 所有关联数据 ✓ 每次操作均写入 audit_logs ✓ 限流基于租户级 RPM 配置
多层防护架构
网络层
k8s NetworkPolicy · Service Type · Port Isolation
接入层
9-Layer Middleware Stack · RBAC · RateLimit
数据层
tenant_id FK · prefix-per-tenant · CASCADE DELETE
存储层
MinIO object lock · Redis AUTH · PG TLS
审计层
audit_logs · 操作溯源 · GDPR Export
设计原则
01 纵深防御 — 每层独立生效, 单点失效不穿透
02 零信任 — 不信赖任何外部输入, 全部来自凭证
03 默认安全 — 隔离优于权限, 权限优于日志
04 可审计 — 所有操作均可溯源到 tenant + user
